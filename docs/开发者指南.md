## 1. 开发环境搭建

### 1.1 依赖版本要求

| 工具 | 最低版本 | 推荐版本 | 备注 |
|------|---------|---------|------|
| JDK | 11 | 11 LTS | 必须 |
| Maven | 3.6 | 3.9 | 必须 |
| MySQL | 8.0 | 8.0.33 | 必须 |
| Redis | 6.0 | 7.x | 开发可选（测试会自动跳过） |
| Node.js | 16 | 18 LTS | 前端必须 |
| HBuilderX | 4.x | 最新 | 小程序开发 |
| RabbitMQ | 3.12 | 最新 | 开发可选（dev 环境已禁用） |

### 1.2 数据库初始化

```bash
# 登录 MySQL
mysql -u root -p

# 创建数据库
CREATE DATABASE ct_tibet_wms CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE ct_tibet_wms;
exit;

# 执行建表 SQL（含初始测试数据）
mysql -u root -p ct_tibet_wms < sql/create_database.sql

# 或使用脚本
cd sql && ./install.sh    # Linux/Mac
cd sql && install.bat     # Windows
```

### 1.3 启动后端（开发模式）

```bash
cd backend

# 开发环境配置（已内置，无需修改）：
# - MySQL: localhost:3306, root/root
# - Redis: localhost:6379（无密码）
# - RabbitMQ: 已禁用
# - JWT_SECRET: 须通过环境变量或在 application-dev.yml 覆盖

# 方式一：设置环境变量后运行
export JWT_SECRET="dev-secret-key-min-32-chars-required-here"
mvn spring-boot:run

# 方式二：在 application-dev.yml 中直接写死（仅限开发）
# jwt:
#   secret: dev-secret-key-min-32-chars-required-here

# 启动成功标志
# ==========================================
#    西藏电信仓库管理系统启动成功!
#    CT Tibet WMS Started Successfully!
#    API文档: http://localhost:48888/doc.html
# ==========================================
```

**API 文档访问：** http://localhost:48888/doc.html（Knife4j UI）

前端代理配置在 `vite.config.js`，开发时 `/api` 请求自动代理到 `http://localhost:48888`。

## 3. 后端核心设计

### 3.1 统一响应格式

所有接口使用 `Result<T>` 包装：

```java
// 成功
Result.success(data)
Result.success(data, "操作成功")

// 失败（抛出 BusinessException 由全局异常处理器捕获）
throw new BusinessException(404, "资源不存在");
throw new BusinessException(1001, "库存不足");

// 分页响应（自动从 Page<T> 转换）
PageResult.of(page)
```

响应 JSON：
```json
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }
}
```

### 3.2 乐观锁设计

库存并发控制是系统最关键的技术点，使用两层乐观锁：

**层一：MyBatis-Plus @Version（用于普通库存增减）**

```java
@Version
private Integer version;  // Inventory.java

// updateById() 会自动在 SQL 添加 version 条件并递增
// UPDATE tb_inventory SET quantity=?, version=version+1
//   WHERE id=? AND version=?
int updated = inventoryMapper.updateById(inventory);
if (updated == 0) {
    throw new BusinessException(500, "库存更新失败，请重试");
}
```

**层二：自定义 SQL + 重试（用于锁定/解锁/提交）**

锁定、解锁、提交不适合使用 `@Version` 插件（需要精确控制 `lockedQuantity`），改用带版本条件的自定义 SQL：

```xml
<!-- InventoryMapper.xml -->
<update id="lockOrUnlockInventory">
    UPDATE tb_inventory
    SET locked_quantity = #{lockedQuantity},
        available_quantity = #{availableQuantity},
        version = version + 1, update_time = NOW()
    WHERE id = #{id} AND version = #{version}
</update>
```

调用方使用 `FOR UPDATE` 悲观锁查出当前记录，执行 CAS，最多重试 3 次：

```java
for (int i = 0; i < 3; i++) {
    Inventory inventory = getInventoryWithLock(warehouseId, materialId); // SELECT...FOR UPDATE
    int updated = inventoryMapper.lockOrUnlockInventory(id, version, newLocked, newAvailable);
    if (updated > 0) return true;
    // version 不匹配，重试
}
throw new BusinessException(500, "锁定库存失败，请重试");
```

### 3.3 事务边界

审批通过时，**锁定库存** + **创建出库单** 必须在同一事务内：

```java
// ApplyServiceImpl.approveApply()
@Transactional(rollbackFor = Exception.class)
public void approveApply(ApprovalDTO dto) {
    // 1. 依次锁定各物资库存（任一失败则回滚已锁定的）
    for (ApplyDetail detail : applyDetails) {
        boolean locked = inventoryService.lockInventory(...);
        if (!locked) {
            // 回滚前面已锁的
            for (int j = 0; j < i; j++) inventoryService.unlockInventory(...);
            throw new BusinessException(1001, "库存不足: ...");
        }
    }
    // 2. 更新申请状态为 APPROVED
    apply.setStatus(ApplyStatus.APPROVED);
    applyMapper.updateById(apply);
    // 3. 自动创建出库单（PENDING_PICKUP，不扣减库存）
    outboundService.createOutboundFromApply(...);
}
```

### 3.4 单号生成规则

| 类型 | 格式 | 示例 |
|------|------|------|
| 入库单 | `RK_{部门编码}_{YYYYMMDD}_{5位流水}` | `RK_WL_20260427_00001` |
| 出库单 | `CK_{部门编码}_{YYYYMMDD}_{5位流水}` | `CK_WL_20260427_00001` |
| 申请单 | `SQ_{部门编码}_{YYYYMMDD}_{5位流水}` | `SQ_SC_20260427_00001` |

流水号使用雪花算法 ID 取模 `% 100000`，确保分布式环境下唯一性。

### 3.5 BaseEntity 公共字段

所有实体继承 `BaseEntity`：

```java
public class BaseEntity {
    private Long id;                  // 自增主键
    private LocalDateTime createTime; // 创建时间（MP 自动填充）
    private LocalDateTime updateTime; // 更新时间（MP 自动填充）
    private Long createBy;            // 创建人 ID
    private Long updateBy;            // 更新人 ID
    private Integer deleted;          // 逻辑删除（0-正常 1-已删除）
}
```

逻辑删除全局配置：
```yaml
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: deleted
      logic-delete-value: 1
      logic-not-delete-value: 0
```

## 4. API 接口规范

### 4.1 认证方式

登录成功后，响应中返回 `token`，后续请求在 Header 中携带：

```http
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

Token 有效期：7200 秒（2小时），可配置。

### 5.2 库存表关键字段说明

```sql
CREATE TABLE tb_inventory (
    quantity          DECIMAL(10,2)  -- 总库存（含锁定）
    locked_quantity   DECIMAL(10,2)  -- 锁定中（申请审批通过，等待取货）
    available_quantity DECIMAL(10,2) -- 可用库存 = quantity - locked_quantity
    version           INT            -- 乐观锁版本号
);
```

库存状态关系：
- `available_quantity = quantity - locked_quantity`
- 审批通过时：`locked_quantity += 申请数量`，`available_quantity -= 申请数量`，`quantity` 不变
- 确认取货时：`quantity -= 取货数量`，`locked_quantity -= 取货数量`，`available_quantity` 不变

### 5.3 关键关联关系

```
tb_apply.outbound_id ←→ tb_outbound.id       （审批通过自动关联）
tb_outbound.apply_id ←→ tb_apply.id
tb_apply.warehouse_id → tb_warehouse.id
tb_warehouse.manager_id → tb_user.id         （仓管员，用于审批权限校验）
tb_user.dept_id → tb_dept.id
tb_inventory.(warehouse_id, material_id)      （联合唯一索引）
```

## 7. 核心业务逻辑

### 7.1 直接出库流程

```
POST /api/outbounds/direct
    ↓
检查库存充足（checkInventory）
    ↓
创建出库单（status=COMPLETED）
    ↓
遍历明细，decreaseInventory（立即扣减，记录库存流水）
    ↓
返回 outboundId
```

### 7.2 申请出库完整流程

**步骤 1：员工提交申请**
```
POST /api/applies
    → 创建 tb_apply（status=PENDING）
    → 创建 tb_apply_detail 明细
```

**步骤 2：仓管员审批通过**
```
POST /api/applies/approve {approvalResult: 1}
    → 依次调用 lockInventory()（乐观锁重试，失败则回滚已锁定的）
    → 更新 apply.status = APPROVED
    → 调用 createOutboundFromApply()
        → 创建 tb_outbound（status=PENDING_PICKUP，source=FROM_APPLY）
        → 创建 tb_outbound_detail
```

**步骤 3：仓管员确认领取**
```
POST /api/outbounds/{id}/confirm
    → 查询明细，调用 commitInventory()（锁定库存转实际扣减）
    → 更新 outbound.status = COMPLETED
    → 更新 apply.status = COMPLETED
```

**超时取消（定时任务）**
```
每天 2:00AM
    → 查询 status=PENDING_PICKUP AND outbound_time <= NOW()-7天 的出库单
    → 调用 cancelOutbound()：unlockInventory + 更新状态 + 更新申请状态
    → 发送通知
```

### 7.3 库存流水记录

每次库存变动自动记录到 `tb_inventory_log`：

| `change_type` | 含义 | `change_quantity` |
|---------------|------|------------------|
| 1 | 入库 | 正数 |
| 2 | 出库 | 负数 |
| 3 | 锁定 | 0（只增加 locked_quantity） |
| 4 | 解锁 | 0（只减少 locked_quantity） |

### 11.2 分布式锁

任务通过 `RedisLockUtils.tryLock()` 防止多实例重复执行：

```java
String lockValue = redisLockUtils.tryLock(lockKey, 300); // 锁定 300 秒
if (lockValue == null) return; // 已有实例在执行
try {
    executeTask();
} finally {
    redisLockUtils.unlock(lockKey, lockValue);
}
```

**开发/测试环境**：`redisLockUtils` 为 `@Autowired(required = false)`，Redis 未启动时注入为 `null`，任务直接执行（跳过锁）。

## 13. 常见问题与调试

### Q1: 启动报 `JWT_SECRET` 相关错误

**原因**：`application.yml` 中 `jwt.secret: ${JWT_SECRET}` 读取环境变量，变量未设置时启动失败。

**解决**：
```bash
# 方式一：环境变量
export JWT_SECRET="dev-secret-key-at-least-32-characters-long"

# 方式二：在 application-dev.yml 中覆盖（仅限开发）
jwt:
  secret: dev-secret-key-at-least-32-characters-long
```

### Q2: 乐观锁冲突频繁导致 500 错误

**原因**：高并发下多个请求同时修改同一库存记录，3 次重试耗尽。

**排查**：查看日志中 `乐观锁冲突，重试第N次` 的频率，考虑增大重试次数或降低并发。

### Q3: 测试通过但生产环境库存数据异常

**原因**：测试使用 H2 内存库，`FOR UPDATE` 行为与 MySQL 不同，可能掩盖并发问题。

**解决**：在测试环境搭建真实 MySQL，使用 `@ActiveProfiles("mysql-test")` 切换。

### Q4: 审批通过后出库单状态显示不对

**排查路径**：
1. 检查 `tb_apply.status` 是否为 `APPROVED`（枚举值 1）
2. 检查 `tb_outbound.apply_id` 是否关联正确
3. 检查 `tb_outbound.status` 是否为 `PENDING_PICKUP`（枚举值 0）
4. 检查 `tb_inventory.locked_quantity` 是否正确增加

### Q5: 前端 v-permission 指令不生效

**原因**：用户 store 的 `roles` 数组格式不匹配（大小写/带前缀 ROLE_ 等）。

**排查**：检查 `store/modules/user.js` 中 roles 如何从接口响应中提取，与 `constants/index.js` 中角色常量对比。

### Q6: H2 测试时 `FOR UPDATE` 报语法错误

**原因**：H2 MySQL 兼容模式不完全支持 `SELECT ... FOR UPDATE`。

**当前处理**：`getInventoryWithLock()` 中的 `wrapper.last("FOR UPDATE")` 在 H2 下会被忽略或报错。测试环境单线程运行，无并发问题，忽略此问题；生产环境使用真实 MySQL，正常工作。
